# 01. 문제 정의 (Problem Definition)

> 이 문서는 전화 기반 수동 운영 방식이 낳는 페인포인트를 분석하고,
> 표면 증상 뒤의 **근본 원인**을 식별해 플랫폼이 해결하는 실질 가치를 정의한다.
>
> **도메인**: BioCom(biocom.kr)은 7종의 검사 서비스(대사 6종·음식물 과민·스트레스/노화·영양/중금속·장내 미생물·호르몬·펫 영양)를 제공하고, 검사 결과 해석 상담을 거쳐 보충제 라인과 **관리 프로그램(챌린지)** 등록까지 이어지는 3-step 비즈니스(검사 → 상담 → 관리 프로그램)를 운영한다. 본 플랫폼은 이 상담·전환 운영을 셀프서비스로 디지털화한다.

---

## 1. 현재 상황 — 전화 기반 수동 프로세스

BioCom 검사 결과를 수령한 고객은 결과 해석과 후속 상품·관리 프로그램 상담을 위해 상담사에게 연락한다.
현재 운영 방식은 **전화·메모 기반의 완전 수동 프로세스**로, 다음과 같이 작동한다.

1. 고객이 유선으로 상담사를 지목하거나 콜센터로 연결 요청
2. 상담사(또는 운영 담당자)가 수기/스프레드시트로 일정 조율
3. 상담 내용·관심 상품·관리 프로그램(챌린지) 등록·결과를 담당자 재량으로 기록(양식 없음)
4. 관리자가 기록을 수동으로 취합해 전환율·상품 관심도·챌린지 등록 전환을 집계

이 방식은 규모가 작을 때는 작동하지만, **상담사 수가 늘거나 동시 요청이 증가하면 즉시 한계가 드러난다.**

## 2. 페인포인트 (Pain Points)

| 영역 | 페인포인트 |
|------|-----------|
| **예약·일정** | 상담사 가용 시간을 고객이 알 수 없어 전화로 개별 확인. 담당자가 달력을 수동 관리하므로 이중 예약·누락이 발생한다. |
| **상담 기록** | 기록 양식이 없어 상담사마다 형태가 다르다. 관심 상품, 전환 outcome, 논의한 검사 지표가 각자 다른 형식으로 남거나 아예 남지 않는다. |
| **전환 집계** | 전환율·상품별 관심도를 관리자가 기록을 읽고 수작업으로 집계한다. 실시간 파악이 불가능하고 오류가 잦다. |
| **가족 대리 상담** | 고객이 가족 검사결과를 대신 상담받으려 할 때 "누구의 결과인지"를 시스템이 추적하지 못해 상담사가 구두로 확인한다. |
| **공급 부족 시 이탈** | 원하는 상담사의 시간대가 꽉 찼을 때 대기 방법이 없다. 고객은 재연락하거나 이탈하고, 수요 정보가 사라진다. |
| **no-show** | 예약 확인·리마인더가 없어 당일 연락 두절이 빈번하다. 상담사의 시간이 낭비되고 일정이 공백으로 남는다. |

## 3. 대상 사용자 (Target Users)

### 3.1 고객 (Customer)

검진 결과를 수령한 일반 사용자.

- 필요: 상담사 가용 시간을 직접 확인하고 예약. 상담 대상을 본인 또는 가족 중 선택.
- 페인: 전화 연결 대기, 이중 예약 사고, 공석 알림 부재.

### 3.2 가족 검사결과 대리 상담자 (Family Proxy)

가족 구성원의 검진 결과를 대신 상담받는 고객.

- 필요: "내 예약이지만 검사 대상은 가족"임을 시스템에 명확히 지정.
- 페인: 구두로 "이건 제 어머니 결과예요"를 설명해야 하고, 기록에 남지 않아 상담사가 혼동.
- 모델링 결정: 가족 관계를 **대칭형 `FamilyLink`(Customer↔Customer)**로 모델링한다. 두 고객 계정을 초대 코드로 연결하고, `ACCEPTED` 링크가 성립하면 서로의 검사결과에 접근할 수 있다. 예약(`Booking`)의 subject는 **항상 `CUSTOMER`**이며, `subjectId`는 예약된 검사결과를 소유한 고객(본인 또는 `ACCEPTED` 가족 파트너)을 가리킨다.

### 3.3 상담사 (Counselor)

건강검진 결과를 해석하고 후속 상품을 안내하는 전문 인력.

- 필요: 본인 일정 조회, 상담 후 주요 상담 내용·권고 사항·후속 조치(구조화 3슬롯)·관심 상품·전환 outcome·상담행위 체크리스트·논의 검사 지표를 구조화된 양식으로 입력. 구매 전환 시 고객을 **관리 프로그램(챌린지)에 등록**(BioCom 3-step의 step-3). 고정 슬롯과 체크리스트로 상담사 간 기록 일관성을 강제한다.
- 페인: 기록 양식 부재로 나중에 통계를 낼 수 없고, 다른 상담사의 일정과 기록을 볼 이유도 없는데 권한 경계가 없다.

### 3.4 관리자 (Admin)

상담 운영 성과를 모니터링하고 서비스 방향을 결정하는 역할.

- 필요: 전환율, 상품별 관심도, 어떤 검사 지표가 전환으로 이어졌는지, **관리 프로그램(챌린지) 등록 수·등록 전환율**(구매 → 챌린지 등록)이 자동 집계된 대시보드.
- 페인: 수작업 집계로 인한 시간 소모·오류·실시간성 결여.

## 4. 근본 원인 분석 (Root-Cause Analysis)

> **이 절이 문제 정의의 핵심이다.**
>
> 요구사항을 나열하기 전에 "표면 증상이 아니라 원인을 봤는가"를 먼저 확인해야 한다.
> 증상만 보고 기능을 대응시키면 실제 이탈 원인(R4)을 Non-goal로 버리거나
> 서비스의 본질인 "검사 기반 상담"(R5)을 데이터 모델에서 누락하는 실수가 생긴다.
> 아래 표는 9개 근본 원인을 증상·원인·대응·배치 순으로 분류한다.

| # | 표면 증상 | 근본 원인 | 대응 | 배치 |
|---|-----------|-----------|------|------|
| **R1** | 수동 예약, 일정 겹침, 누락 | 예약 조정이 사람 간 비동기 통신에 의존 — 원자적 상태 변경을 보장하는 시스템이 없음 | 셀프서비스 예약 + DB 동시성 보장(unique constraint + 트랜잭션) | **MVP (구현)** |
| **R2** | 상담 기록 제각각, 전환율 수동 집계 | 기록 스키마 부재 → 데이터가 구조화되지 않아 집계 불가 | 구조화 기록(`ConsultationRecord`) + 자동 집계(`Analytics`) | **MVP (구현)** |
| **R3** | 가족 검사결과 대리 상담 시 혼선 | 상담 대상(subject)이 예약 계정 소유자와 다를 수 있다는 개념이 모델에 없음 | 대칭형 `FamilyLink`(Customer↔Customer) — `ACCEPTED` 링크로 가족 검사결과 접근. subject는 항상 `CUSTOMER`이고 `subjectId`가 검사결과 소유 고객을 가리킴 | **MVP (구현)** |
| **R4** | 꽉 찬 시간대에서 고객이 대안 없이 이탈 | 공급(상담사 시간) 희소 + 수요 집중 → 대기 수요를 포착하는 메커니즘 없음 | `Waitlist` 엔티티 + 예약 취소 시 FIFO 공석 알림(`SLOT_OPENED`) | **MVP (구현, 단순형)** |
| **R5** | "검사 결과 기반" 상담인데 전환이 결과와 분리됨 | 기록이 *어떤 지표*를 논의했는지 캡처하지 않음 → 전환을 검사 결과에 귀속할 수 없음 | `ConsultationRecordMetric`으로 기록 ↔ 지표 연결 → 지표별 전환 집계 | **MVP (구현)** |
| **R6** | 당일 연락 두절(no-show) | 잊어버림 + 변경 마찰 — 리마인더 없고 reschedule이 어려움 | 리마인더 알림(시뮬레이션) + reschedule 플로우 설계 | 리마인더=MVP / reschedule=**Phase 2 (설계)** |
| **R7** | 이탈의 더 깊은 원인 — 동기 상담만 존재 | 상담사와 실시간으로 만나지 않아도 되는 비동기 대안 채널이 없음 | 비동기 상담 대안(결과지 FAQ · 텍스트 Q&A) | **Phase 2 (설계)** |
| **R8** | waitlist의 지능화 미비 | 단순 선착순 대기 → 선호 상담사·시간대 최적 매칭이 없음 | waitlist 자동 매칭(상담사·시간 최적 추천) | **Phase 2 (설계)** |
| **R9** | 구매 전환이 관리 프로그램 등록(step-3)으로 이어졌는지 추적 불가 | BioCom 3-step(검사→상담→관리 프로그램)의 step-3이 데이터 모델에 없음 — 상담사가 챌린지 등록을 구두/수기로만 처리해 "구매가 실제 관리 프로그램 등록으로 이어졌는가"를 측정할 수 없음 | `Challenge`(카탈로그) + `ChallengeEnrollment`(상담 기록 생성 시 상담사 등록) + 챌린지 등록 전환율 집계 | **MVP (구현)** |

### 원인 분석이 요구사항 결정을 어떻게 바꿨는가

초안 설계에서는 **R4(공급 희소로 인한 이탈)**를 전체 Non-goal로 처리하고,
**R5(전환과 검사 지표의 분리)**를 데이터 모델에서 누락했다.

이는 증상("꽉 찼을 때 고객이 떠난다")은 인식했지만 원인("대기 수요를 포착하는 메커니즘이 없다")은 보지 못한 결과다.
근본 원인 분석을 통해 두 항목을 MVP로 끌어올렸다:

- **R4 대응**: `Waitlist` 엔티티와 취소 시 FIFO 알림을 최소 구현(데이터 모델 + API)으로 포함.
  자동 매칭(R8)은 별도 지능화 문제이므로 Phase 2 설계로 분리.
- **R5 대응**: `ConsultationRecordMetric` 조인 엔티티로 상담 기록과 검사 지표를 연결하고,
  Analytics에서 "지표별 전환율"을 집계해 검사 기반 서비스의 본질을 데이터로 증명.

R6·R7·R8은 원인을 인식하고 있으나, 2주 솔로 일정 제약과 설계역량 증명 우선 원칙에 따라
MVP 구현이 아닌 Phase 2 설계(인터페이스 경계 + 상태전이 명시)로 배치한다.
"원인을 알고 근거 있게 미룬 것"이 단순 미구현과 다른 점이다.

- **R9 대응(BioCom finalization)**: 검사 → 상담 → 관리 프로그램(챌린지)으로 이어지는 BioCom 3-step의 step-3을
  `Challenge`(시드 관리 카탈로그) + `ChallengeEnrollment`(상담사가 상담 기록 생성 시 등록)로 모델링하고,
  Analytics에 챌린지 등록 수와 **등록 전환율(구매 기록 → 챌린지 등록)**을 추가해 "상담이 관리 프로그램 등록으로
  이어졌는가"를 데이터로 증명한다. 등록은 상담 기록 생성 트랜잭션 안에서 원자적으로 발생한다(설계 근거는 ADR 0007).

## 5. 해결 가치 (Value Proposition)

| 대상 | 현재 상태 | 플랫폼 이후 |
|------|----------|------------|
| 고객 | 전화 연결 → 일정 조율 → 결과 구두 설명 | 슬롯 조회 → 즉시 예약 → subject(본인/가족) 지정 → 확인 알림 수신 |
| 가족 대리 상담자 | 구두로 "가족 결과"임을 설명, 기록 불안정 | `ACCEPTED` `FamilyLink`로 가족 검사결과를 상담 대상으로 지정, 소유권 검증으로 오지정 방지 |
| 상담사 | 구두 일정 확인 + 자유 형식 기록(상담사마다 형태 제각각) | 본인 일정 즉시 조회 + 구조화 양식(주요 상담 내용·권고 사항·후속 조치 3슬롯 + 상담행위 체크리스트 + 관심상품·outcome·검사지표)으로 입력 + 구매 전환 시 관리 프로그램(챌린지) 등록 → 고정 슬롯·체크리스트로 기록 일관성 강제 |
| 관리자 | 기록 수동 취합 → 지연·오류 빈발 | 실시간 집계 대시보드(전환율 · 상품별 관심 · 지표별 전환 · **챌린지 등록 수·등록 전환율**) |
| 서비스 전체 | 공석 수요 소실, 이탈 측정 불가 | Waitlist로 수요 포착 → 취소 발생 시 자동 공석 알림 |
